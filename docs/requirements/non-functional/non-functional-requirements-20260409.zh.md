# 非功能需求规格说明书

**项目名称**: 点数换算小工具
**创建日期**: 2026-04-09
**版本**: 1.0

---

## NFR-001: 性能要求

### 响应时间

- **交互响应**: 用户的每次点击或输入，界面更新延迟应 < 100ms。
- **计算速度**: 输入参数改变后，实时点数计算的完成时间应 < 50ms。

### 吞吐量

- **并发处理**: 作为纯前端应用，所有计算均在客户端浏览器完成，无需考虑服务器并发压力。
- **系统资源占用**: 应用在运行时的 CPU 占用率和内存消耗应保持在低水平，确保在普通办公电脑或移动设备上也能流畅运行。

### 测量方法

- **浏览器性能分析工具**: 使用 Chrome DevTools 的 Performance 面板测量交互和计算的执行时间。

---

## NFR-002: 可用性与可靠性要求

### 可用性

- **目标**: 由于是独立的前端工具，可用性主要依赖于用户的浏览器和设备环境。
- **降级处理**: 如果某些非核心资源加载失败，应用的核心计算功能仍应能正常工作。
- **容错性**: 对于异常输入（如非数字字符、极大数值、除数为零等），应用不应崩溃，并给出明确的错误提示。

### 可靠性

- **无网络运行**: 在初始加载完成后，即使断开网络连接，应用的各项计算功能也必须能够完全正常运作。

---

## NFR-003: 安全性要求

### 概述

作为一个纯前端、无后端的公开工具，它不处理任何敏感的个人信息或商业机密数据。

### 数据隐私

- **本地存储**: 不会在用户的本地存储（如 LocalStorage、Cookies）中保存任何输入的参数数据。每次刷新页面即重置为初始状态。
- **数据传输**: 计算过程完全在本地进行，没有任何数据会被发送到任何外部服务器。

### 访问控制

- **无认证**: 不需要用户登录即可使用该工具的所有功能。
- **权限管理**: 不区分管理员和普通用户角色。

---

## NFR-004: 可扩展性与维护性要求

### 架构设计

- **模块化**: 代码结构应保持清晰的模块化，将 UI 组件、状态管理和核心计算逻辑分离。
- **配置化**: 模型列表数据（包含模型名称、默认输入单价、默认输出单价）应当从一个易于修改的配置文件（如 JSON 文件）中读取，以便将来快速添加或修改模型，而无需更改核心计算代码。
- **公式可配置**: 核心的计算公式逻辑应该被封装成独立的函数，如果未来需要引入更复杂的计费公式，只需修改或替换该函数即可。

### 部署与发布

- **静态部署**: 该工具应该能够作为一组静态文件（HTML/CSS/JS）部署到任何标准的 Web 服务器、CDN 或静态站点托管服务（如 GitHub Pages、Vercel 等）。
